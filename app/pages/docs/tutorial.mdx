Dans ce tutoriel, nous allons vous apprendre à créer use simple
application de vote.

Ce guide suppose que vous avez déjà [installé Blitz](./get-started). Vous
pouvez vérifier si c'est le cas ainsi que quelle version vous avez en
exécutant la commande suivante dans votre terminal :

```sh
blitz -v
```

Si Blitz est bien installé, vous devriez voir le numéro de la version de
votre installation. Sinon il ne l'est pas, vous devriez voir une erreur
comme "command not found: blitz".

## Créez une nouvelle application {#creating-a-new-app}

Depuis le terminal, déplacez-vous dans le répertoire où vous souhaitez
créer votre application en utilisant la commande `cd`. Ensuite, exécutez
la commande suivante :

```sh
blitz new mon-appli-blitz
```

Blitz s'occupera de créer un dossier `mon-appli-blitz` dans le répertoire.
Il vous sera également demandé de choisir un module de gestion de
formulaire. Pour les exemples suivant, choisissez le module recommandé
`React Final Form`.

Jettons un oeil à ce que `blitz new` a créé :

Ces fichiers sont :

- Le répertoire `app/` contient la majorité de votre projet. C'est ici que
  vous ajouterez vos pages et routes d'API.

- Le répertoire `app/pages/` est le principal endroit ou vous créerez vos
  pages. Si vous avez déjà utilisé Next.js, vous remarquerez immédiatement
  la différence. Avec Blitz, vous pouvez avoir plusieurs répertoires
  `pages` qui seront combinés au moment de la compilation.

- Le répertoire `app/core/` est l'endroit parfait pour stocker vos
  composants, hooks et autre code réutilisable à travers votre
  application.

- Le répertoire `db/` contient la configuration de votre base de données.
  Si vous voulez créer un nouveau modèle ou lancer une migration, c'est
  ici que vous trouverez ce qu'il vous faut.

- Le répertoire `public/` contient toutes vos resources statiques. Si
  votre application nécéssite des images, documents ou vidéos, c'est ici
  que vous devriez les stocker.

- Les fichiers `.babelrc.js`, `.env`, etc. aussi connus sous le nom de
  "dotfiles", sont des fichiers de configuration pour différents outils
  JavaScript.

- `blitz.config.js` offre aux utilisateurs avancés la possibilité de
  personnaliser leur installation de Blitz. Il supporte le même format que
  `next.config.js`.

- `tsconfig.json` contient nos règles de configuration préférées pour le
  compilateur TypeScript.

## Configurez une base de données {#database-setup}

Bonne nouvelle! Une base de données SQLite est déjà configurée! Vous

Veuillez notez que dans le contexte d'un vrai projet, vous auriez avantage

## Générez le code de vos modèles {#scaffolding-code-for-our-models}

Blitz vous permet de générer le code nécéssaire pour une multitude de
situations via la commande [`generate`](./cli-generate). Nous allons
l'utiliser afin de générer deux modèles : `Question` et `Choice`. Une
`Question` consiste du texte de la question ainsi qu'une liste de choix de
réponse. Un `Choice` contient le texte du choice, un nombre de votes et la
question à laquelle il fait référence. Blitz s'occupe de générer un `id`
unique, la date de création ainsi que la date de la dernière modification
pour chacun des modèle.

#### Premièrement, nous allons générer le modèle `Question` :

```sh
blitz generate all question texte:string
```

`prisma migrate` pour mettre votre schema de base de données à jour avec

  const goToNextPage = () => {
    if (!isPreviousData && hasMore) {
      router.push({query: {page: page + 1}})
    }

  return (
    <div>
      <ul>

### `deleteQuestion` mutation {#delete-question-mutation}

[//]: # "Remove this section once Prisma supports cascading deletes"

Prisma does not yet support "cascading deletes". In the context of this
tutorial, that means it does not currently delete the `Choice` data when
deleting a `Question`. We need to temporarily augment the generated
`deleteQuestion` mutation in order to do this manually. Open up
`app/questions/mutations/deleteQuestion.ts` in your text editor and add
the following to the top of the function body:

```ts
await db.choice.deleteMany({ where: { questionId: id } })
```

The end result should be as such:

```diff
// app/questions/mutations/deleteQuestion.ts

export default resolver.pipe(
  resolver.zod(DeleteQuestion),
  resolver.authorize(),
  async ({id}) => {
+   await db.choice.deleteMany({where: {questionId: id}})
    const question = await db.question.deleteMany({where: {id}})

    return question
  },
)
```

This mutation will now delete the choices associated with the question
prior to deleting the question itself.

## Listing choices {#listing-choices}

Time for a breather. Go back to `localhost:3000/questions` in your browser
and look at all the questions you‘ve created. How about we list these
questions’ choices here too? First, we need to customize the question
queries. In Prisma, you need to manually let the client know that you want
to query for nested relations. Change your `getQuestion.ts` and
`getQuestions.ts` files to look like this:

```diff
// app/questions/queries/getQuestion.ts

const GetQuestion = z.object({
  // This accepts type of undefined, but is required at runtime
  id: z.number().optional().refine(Boolean, "Required"),
})

export default resolver.pipe(
  resolver.zod(GetQuestion),
  resolver.authorize(),
  async ({id}) => {
-   const question = await db.question.findFirst({where: {id}})
+   const question = await db.question.findFirst({
+     where: {id},
+     include: {choices: true},
+   })

    if (!question) throw new NotFoundError()

    return question
  },
)
```

```diff
// app/questions/queries/getQuestions.ts

interface GetQuestionsInput
  extends Pick<
    Prisma.QuestionFindManyArgs,
    "where" | "orderBy" | "skip" | "take"
  > {}

export default resolver.pipe(
  resolver.authorize(),
  async ({where, orderBy, skip = 0, take = 100}: GetQuestionsInput) => {
    const {items: questions, hasMore, nextPage, count} = await paginate({
      skip,
      take,
      count: () => db.question.count({where}),
      query: (paginateArgs) =>
        db.question.findMany({
          ...paginateArgs,
          where,
          orderBy,
+         include: {choices: true},
        }),
    })

    return {
      questions,
      nextPage,
      hasMore,
      count,
    }
  },
)
```

## Lastly, let's allow editing choices for an existing question {#edit-choices-for-question}

If you click the **Edit** button on one of your existing questions, you'll
see it uses the same form as creating questions. So that part is already
done! We only need to update our mutation.

Open `app/questions/mutations/updateQuestion.ts` and make the following
changes:

```diff
// app/questions/mutations/updateQuestion.ts
import {resolver} from "blitz"
import db from "db"
import * as z from "zod"

const UpdateQuestion = z
  .object({
    id: z.number(),
    text: z.string(),
+   choices: z.array(
+     z.object({id: z.number().optional(), text: z.string()}),
+   ),
  })

export default resolver.pipe(
  resolver.zod(UpdateQuestion),
  resolver.authorize(),
  async ({id, ...data}) => {
-   const question = await db.question.update({where: {id}, data})
+   const question = await db.question.update({
+     where: {id},
+     data: {
+       ...data,
+       choices: {
+         upsert: data.choices.map((choice) => ({
+           // Appears to be a prisma bug,
+           // because `|| 0` shouldn't be needed
+           where: {id: choice.id || 0},
+           create: {text: choice.text},
+           update: {text: choice.text},
+         })),
+       },
+     },
+     include: {
+       choices: true,
+     },
+   })

    return question
  },
)
```

[`upsert`](https://www.prisma.io/docs/reference/api-reference/prisma-client-reference#upsert)
is a special operation that means, "If this item exists, update it. Else
create it". This is perfect for this case because we didn't require the
user to add three choices when creating the question. So if later the user
adds another choice by editing the question, then it'll be created here.
